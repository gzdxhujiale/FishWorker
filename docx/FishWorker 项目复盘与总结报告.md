# FishWorker 项目复盘与总结报告

## 1. 项目概况与背景

FishWorker 是一个个人效能管理桌面端应用（基于 Tauri + React）。本项目的一大特色在于**高度依赖 AI Coding / Vibecoding 模式驱动**，所有的业务逻辑、UI 界面均通过细致的 Markdown 需求文档和接口文档交由 AI 编写完成。模块涵盖了知识库、四象限工作台、周计划、清单系统等。

## 2. 成功经验

### 2.1 高效的“搭积木”式模块化扩展

得益于早期建立的 `Frontend_Interfaces.md` 规范，应用形成了一个高度解耦的架构（如 `ToolConfig` 接口定义了统一的侧边栏工具挂载方式）。这使得在引入“四象限工作台”或“清单系统”时，AI 能够像插入插件一样，非常顺利地完成新功能的集成，几乎没有对现有系统的破坏。

### 2.2 文档作为最强大的 Context

本项目积累了大量详实的 PRD（如 `Four_Quadrants_Workdesk_PRD.md`、`time_management_prd_v1.1.md`）。这些文档不仅包含了功能描述，更重要的是包含了：

- **边界定义**（什么不做）
- **数据结构定义**（TypeScript Interface 示例）
- **视觉及组件树拆解** 这种“保姆级”的结构化输入，大幅减少了 AI 的幻觉，保证了代码生成的质量和一次性通过率。

### 2.3 小步快跑与独立重构的节奏

项目没有试图在一个版本中解决所有问题，而是体现出了很好的迭代意识。例如时间管理模块从初始版本进化到 `v1.1`，专门针对“隐藏已完成任务”、“增加截止时间”出具了增量 PRD；针对状态管理的痛点，专门出具了《Zustand 迁移规划》，确保了业务迭代与技术优化的分离。

## 3. 面临的挑战与技术债务

### 3.1 早期“随性 AI 编码”遗留的状态管理危机

在项目的早期开发中，由于过分追求实现速度，AI 生成了大量基于 `window.dispatchEvent`、`addEventListener` 和手动 `refreshData()` 的状态同步代码。

- **痛点**：随着功能的丰富（特别是重度拖拽交互的引入），这种模式导致了严重的性能瓶颈（全量重绘卡顿）以及潜在的内存泄漏风险。
- **解决**：后期不得不专门产出 `Zustand_Migration_PRD.md` 来进行全局的状态库重构，将数据流收敛为单向、细粒度的 Selector 订阅模式。

### 3.2 复杂交互场景下的“幻觉”

在涉及复杂的 HTML5 拖拽（Drag & Drop）和绝对定位弹窗（如 `QuickAddPopover`）时，AI 有时会生成看似合理但存在边界 Bug 的代码。

- **痛点**：复杂的 UI 交互逻辑很难完全用自然语言无死角地描述。
- **解决**：在 PRD 中明确指定底层技术方案（如：“使用原生 DOM 边界计算定位”，“使用原生 HTML5 Drag/Drop API”），减少 AI 自由发挥的空间。

## 4. 下一步行动与建议

1. **提前锁定核心基建 (Framework First)** 在未来的 AI 编程项目中，**第一步不应是写业务 PRD，而是写基建 PRD**。在项目 Day 1 就必须由人类架构师/主导者确定好状态管理框架（如 Zustand）、路由方案、样式系统（如 Tailwind/CSS Module），并生成相应的规范文档。
2. **规范化 Prompt 模板** 后续开发新功能时，应固定输入模板：“`全局 Interface 规范文档` + `本次 Feature PRD` -> `AI 生成代码`”。通过注入全局规范，保证代码风格不跑偏。
3. **文档即代码 (Docs as Code)** 项目的 PRD 是项目最重要的资产之一，甚至比代码更重要。当代码逻辑发生调整时，必须同步更新 PRD（特别是数据结构和组件列表），保持文档库永远是 Source of Truth，这样未来接手的 AI 依然有上下文可依。
