# Product Requirements Document (PRD): 习惯管理功能

## 1. 概述

- **问题陈述：** 用户难以追踪和坚持日常习惯，缺乏一个直观且能够提供激励反馈的工具来帮助他们建立规律的生活方式。
- **建议的解决方案：** 在我们的应用中引入一个“习惯”管理模块，允许用户创建自定义习惯，设定频率和目标，并通过每日打卡、数据统计（如连续打卡天数）和视觉反馈来激励用户坚持。

## 2. 核心理念

习惯养成的核心在于一致性和正向反馈。本功能旨在通过简化记录过程（一键打卡）和可视化进度（日历视图、连胜记录），降低用户坚持习惯的心理门槛，将“坚持”转化为一种具有成就感的游戏化体验。

## 3. 解决方案概述

### 3.1 建议的解决方案

- **高层级描述**: 构建一个习惯追踪系统，包括习惯列表展示、打卡操作、习惯详情与统计，以及习惯的创建和编辑功能。
- **核心能力**:
  - 习惯卡片的创建、编辑与分组管理。
  - 每日/多频率习惯的快捷打卡机制。
  - 习惯完成情况的数据统计与可视化展示（日历、连续天数）。

### 3.2 包含在本次范围内

- **功能点 1：习惯主页与列表展示** (高优先级)
  - 以周视图（包含今日）展示近期日期。
  - 按分组（如：上午、下午等）展示用户的习惯卡片列表。
  - 习惯卡片上需显示习惯名称、图标、当前连续打卡天数及总打卡天数。
  - 提供直接在列表页对当天习惯进行打卡的快捷操作（复选框/圆圈）。
- **功能点 2：习惯详情与统计看板** (高优先级)
  - 点击习惯卡片弹出详情侧边栏/浮层。
  - 展示关键统计数据：月打卡天数、总打卡天数、月完成率、当前连续天数。
  - 提供当月打卡日历视图，直观显示每天的完成情况。
  - 包含对该习惯的操作菜单：编辑、开始专注（可选扩展）、打卡样式设置、归档、删除。
- **功能点 3：习惯创建与编辑** (高优先级)
  - 提供弹窗进行习惯的详细设置。
  - 可设置属性：习惯名称、频率（如：每天、每周）、目标（当天完成打卡、当天完成一定量打卡）、开始日期、坚持天数（如：永远、7天、21天、自定义）、所属分组（上午、下午、晚上、其他、自定义分组）。
  - 支持设置定时提醒时间。
  - 支持开启/关闭“自动弹出打卡日志”选项。

### 3.3 超出本次范围

- 复杂的社交分享与好友排行榜功能（未来迭代）。
- 与外部日历或健康数据的深度整合。
- 复杂的自定义数据分析报表导出。

## 4. 用户故事与需求

### 4.1 用户故事

**用户故事 1：快捷打卡**

> 作为一名 想要养成早起习惯的用户 我希望 能在应用首页看到我今天的习惯列表并直接点击完成 以便于 我能快速记录我的行为，不需要繁琐的步骤。
>
> 验收标准： [ ] 首页显示当天的习惯列表。 [ ] 习惯卡片右侧有明显的打卡操作区域（如空心圆圈）。 [ ] 点击后状态即时更新为已完成（如实心状态或打勾），并有视觉反馈。

**用户故事 2：查看进度与成就感**

> 作为一名 正在坚持每天阅读的用户 我希望 能看到我这个月到底坚持了多少天，以及我最长连续坚持了多久 以便于 我能从数据中获得成就感，激励自己继续坚持。
>
> 验收标准： [ ] 点击习惯能进入详情页。 [ ] 详情页清晰展示“总打卡”、“月完成率”、“当前连续”等数据卡片。 [ ] 详情页提供月度日历视图，标记出已打卡的日期。

**用户故事 3：灵活定制习惯**

> 作为一名 有着特定作息规律的用户 我希望 能够自定义我的习惯，比如设置只在工作日提醒我，并把它们分类到“上午”或“晚上” 以便于 习惯列表符合我的真实生活节奏，不会显得杂乱。
>
> 验收标准： [ ] 提供创建/编辑习惯的入口。 [ ] 能够自定义习惯名称、图标。 [ ] 能够设置频率、所属分组。 [ ] 能够设置特定时间的提醒。

### 4.3 非功能需求

- **性能 (Performance)：** 打卡操作的响应时间必须在 200ms 以内，确保流畅体验。列表加载时间应小于 1s。
- **可扩展性 (Scalability)：** 数据库设计需考虑用户产生的大量打卡日志数据，支持按月/年高效查询统计。
- **易用性 (Usability)：** 界面设计需保持极简，打卡操作目标区域需足够大，方便触控操作（如果是移动端优先或响应式）。

## 5. 设计与用户体验

### 5.1 设计原则

- **极简且直观：** 核心操作（打卡）必须一触即达，无需进入多层级页面。
- **正向激励：** 状态变化（未完成到已完成）和数据展示（连续天数）需有明显的视觉强化，给予用户奖励感。
- **清晰的信息层级：** 列表页重在操作和当前状态概览，详情页重在历史数据和设置。

### 5.2 页面结构与交互流程

1. **习惯主页：**
   - 顶部：周日历导航（显示前几天、今天、后几天）。
   - 主体：按分组（如“上午”）折叠/展开的习惯列表。
   - 习惯卡片：左侧（图标、名称、简要统计），右侧（一周打卡状态小圆点指示器）。
2. **习惯详情/侧边栏：**
   - 顶部：习惯名称及操作菜单（...）。
   - 数据区：四个关键数据卡片（月打卡、总打卡、月完成率、当前连续）。
   - 日历区：当月日历，标记打卡状态。
3. **编辑/创建弹窗：**
   - 表单结构，包含：图标/名称输入、频率选择、目标设置、开始日期、持续时间、分组选择、提醒设置开关。

### 5.3 信息架构

- 用户 (User)
  - 习惯分组 (Category)
    - 习惯实例 (Habit)
      - 打卡记录 (Check-in Log)
      - 统计数据 (Statistics)

## 6. 技术规格说明

### 6.1 API 设计 (草案)

- `GET /api/habits`: 获取用户的习惯列表及当日状态。
- `POST /api/habits`: 创建新习惯。
- `PUT /api/habits/:id`: 更新习惯设置。
- `DELETE /api/habits/:id`: 删除或归档习惯。
- `POST /api/habits/:id/checkin`: 执行打卡/取消打卡操作 (需包含日期参数)。
- `GET /api/habits/:id/stats`: 获取特定习惯的详细统计数据和历史日历数据。

### 6.2 数据库设计 (关键实体)

- **Habit (习惯表):** `id`, `user_id`, `name`, `icon`, `frequency_rule`, `category_id`, `created_at`, `status` (active/archived)。
- **CheckinLog (打卡记录表):** `id`, `habit_id`, `user_id`, `date`, `status` (completed/skipped), `created_at`。 (注意：由于打卡是高频操作，建议记录每一天的明确状态，方便统计计算)。
