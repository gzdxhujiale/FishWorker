# 产品需求文档 (PRD)：每日复盘～复利计划 (Daily Review & Compound Interest)

## 1. 概述
### 包含:
- **问题陈述：** 用户在日常工作和学习中缺乏结构化的回顾机制，容易导致经验无法沉淀；同时，每天微小的努力和进步难以量化，导致缺乏长期坚持的动力和正向反馈。
- **建议的解决方案：** 在 FishWorker 中新增“每日复盘～复利计划”页面功能。提供无压力的轻量级每日复盘记录功能（增删改查），并引入“复利”概念可视化每日坚持的价值，激励用户持续记录，形成个人成长的飞轮。

## 2. 核心理念

- **复利效应 (Compound Interest Effect)**：每天进步 1%，一年后将提升近 38 倍 ($1.01^{365} \approx 37.78$)。反之，每天退步 1%，一年后将趋近于 0。通过将用户的连续复盘行为转化为可视化的“复利曲线”或“能量值”，让坚持看得见。
- **PDCA闭环融合**：系统现有的 `time-management`（计划与执行）和 `unlearning-loop`（解构与学习）模块偏重于“做”。复盘模块则补齐了 Check & Act 环节，让整个 FishWorker 成为完整的个人操作系统。

## 3. 解决方案概述

### 3.1 建议的解决方案
- **高层级描述**: 一个结合了日志记录与游戏化成长的独立功能页面。
- **核心能力**: 
  - 每日复盘编辑器（富文本/Markdown支持）。
  - 历史复盘的日历视图及时间轴流。
  - 复利成长数据看板（连续打卡热力图、复利增值曲线）。

### 3.2 包含在本次范围内
- **复盘记录 (CRUD)：** 创建今日复盘、查看历史复盘、修改过往复盘、删除无效内容。
- **复利看板：** 展示打卡日历（类似 GitHub 贡献热力图），计算连续打卡天数（Streak），以及基于公式生成的“当前复利值”。
- **本地数据管理：** 基于 Tauri 的本地存储方案保存复盘数据。

### 3.3 超出本次范围
- **社交分享：** 将复盘内容分享到微信、微博等外部平台。
- **AI 智能总结：** 利用大模型自动分析一周或一月的复盘并给出成长建议（留作 V2 迭代）。
- **多端云同步：** 复杂的跨设备实时同步逻辑。

## 4. 用户故事与需求

### 4.1 用户故事
```
作为一名 寻求自我提升的 FishWorker 用户
我希望 能够快速地记录每天的得失和感悟
以便于 沉淀经验，避免重复犯错

作为一名 难以坚持写日记的用户
我希望 看到我连续记录带来的“复利值”增长和热力图
以便于 激励我每天花 5 分钟完成复盘打卡

验收标准：
[ ] 用户可以点击一个按钮立即开始编写当天的复盘。
[ ] 用户可以在历史列表中找到过去某天的复盘并进行修改。
[ ] 页面上有明显的连续打卡天数和热力图展示。
```

### 4.2 功能需求
| ID | 需求描述 | 备注 |
|----|------------|-------|
| FR1 | 用户可以针对每一天创建、编辑、保存每日复盘记录 | 核心能力 (Critical) |
| FR2 | 用户可以删除已创建的复盘记录 | 核心能力 (Critical) |
| FR3 | 提供日历视图，标记出哪些天已经完成了复盘 | 重要 (Important) |
| FR4 | 系统自动计算当前连续复盘天数 (Current Streak) 和历史最高连续天数 (Longest Streak) | 重要 (Important) |
| FR5 | 系统根据复盘天数，通过公式可视化一条“复利成长曲线” | 加分项 (Nice to have) |

### 4.3 非功能需求
- **性能 (Performance)：** 列表页与日历页加载需在秒级内完成，保障无缝体验。
- **可扩展性 (Scalability)：** 数据结构需具备扩展性，以便未来挂载“标签”、“心情”或“AI 分析结果”等字段。
- **安全性与隐私 (Security)：** 所有复盘日记默认保存在本地，绝不上报云端，保障用户最私密的日记安全。

## 5. 设计与用户体验

### 5.1 设计原则
- **极简无压 (Frictionless)**：输入框设计要开门见山，减少用户的思考成本。可以提供默认的复盘模板（如：今天做好了什么？哪里可以改进？明天最重要的一件事？）。
- **正向反馈 (Positive Reinforcement)**：完成每日保存时，提供微交互动画（如撒花、进度条上涨），强化复利积累的爽感。

### 5.2 页面结构与交互流程
整个页面分为左右或上下两部分布局：
- **统计面板区（视觉焦点）**：
  - 左侧或顶部：展示当前的“复利值”（如：1.01的N次方结果）、连续天数、以及 GitHub 风格的年度打卡热力图。
- **复盘工作区（操作焦点）**：
  - 右侧或底部：包含一个日历选择器和内容编辑器。选择今天即进入编辑模式；选择历史日期即可查看或修改。
  - 默认展示当前日期的空白/已填内容。

### 5.3 信息架构
- 页面导航：在侧边栏新增 `[每日复盘]` 入口。
- 数据内容：每篇复盘包括：`日期 (Date)`、`正文 (Content)`、`心情/状态评分 (Rating)`。

## 6. 技术规格说明

### 6.1 核心数据接口设计 (基于本地 Store)
- `getReviewByDate(date: string): DailyReview | null`
- `getAllReviews(): DailyReview[]`
- `saveReview(review: DailyReview): void`
- `deleteReview(date: string): void`
- `getCompoundStats(): { currentStreak: number, maxStreak: number, totalDays: number }`

### 6.2 数据库/本地存储设计
可采用现有的本地文件存储机制（如 JSON 结构存储在应用数据目录）：
```typescript
interface DailyReview {
  id: string;            // UUID
  date: string;          // YYYY-MM-DD
  content: string;       // 复盘正文内容（Markdown 或 HTML）
  rating?: number;       // 当天状态评分 (1-5)
  createdAt: number;     // 时间戳
  updatedAt: number;     // 时间戳
}

interface UserStats {
  lastReviewDate: string; 
  currentStreak: number;
  longestStreak: number;
  baseValue: number;     // 初始值为 1.00，每天打卡基于此复利增加
}
```
