# Product Requirements Document (PRD): 每日复盘同步 TiDB 数据库

## 1. 概述
- **问题陈述：** 目前“每日复盘”模块的数据可能仅存储在本地（或未完全与云端数据库打通），在多设备切换或系统异常时面临数据丢失风险，且无法与其他设备保持状态一致。
- **建议的解决方案：** 参考“知识库”模块的同步机制，为“每日复盘”功能引入数据同步机制，将复盘数据（包括总结、待办、心得等）可靠地同步到远端 TiDB 数据库中，实现云端备份与多端一致性。

## 2. 核心理念
- **数据安全与可靠性：** 用户的复盘数据是高价值的个人数字资产，必须保证其不丢失且随时可追溯。
- **无缝体验：** 同步过程应尽可能对用户透明，在保证性能的前提下，实现自动保存和后台同步，辅以必要的网络异常提示。

## 3. 解决方案概述

### 3.1 建议的解决方案
- **高层级描述**: 为“每日复盘”模块增加 TiDB 云端同步能力，使其具备云端持久化存储功能。
- **核心能力**:
  - 本地与 TiDB 数据库的双向同步/拉取。
  - 自动触发（如：防抖定时保存、切换页面保存）和手动触发同步机制。
  - 同步状态的 UI 提示（如：同步中、已同步、离线/失败）。

### 3.2 包含在本次范围内
- **功能点 1：** 后端 API 接口支持（创建、更新、查询每日复盘记录到 TiDB）。优先级：高。
- **功能点 2：** 前端数据层改造，集成与“知识库”类似的本地+远程同步逻辑，处理网络请求和状态管理（如 `dailyReviewStore`）。优先级：高。
- **功能点 3：** UI 增加同步状态指示器，实时反馈保存状态。优先级：中。
- **功能点 4：** 断网重连或应用重新打开时的全量或增量状态同步逻辑（首次拉取）。优先级：高。

### 3.3 超出本次范围
- 细粒度的字段级冲突解决（如采用 CRDT 等复杂并发控制），初期可以整体覆盖或时间戳后到优先。
- 历史版本回滚与版本历史查看。
- 多人协同编辑复盘内容。

## 4. 用户故事与需求

### 4.1 用户故事
```text
作为一名 使用每日复盘的用户
我希望 系统能在后台自动将我的复盘数据同步到云端 TiDB
以便于 我在更换设备或重装系统后，依然能无缝找回所有复盘记录，不必担心数据丢失。

验收标准：
[ ] 当我在复盘编辑器中停止输入一段时间后，数据自动保存至 TiDB。
[ ] UI 上能清晰看到“已保存至云端”的提示。
[ ] 在另一台设备登录或清空本地缓存后，重新打开复盘页面能拉取到 TiDB 中的最新数据。
```

### 4.2 功能需求
| ID | 需求描述 | 备注 |
|----|------------|-------|
| FR1 | 系统提供 API 端点用于将每日复盘数据持久化至 TiDB | 必须 |
| FR2 | 系统提供 API 端点用于从 TiDB 拉取每日复盘数据 | 必须 |
| FR3 | 前端实现防抖保存（Debounce Save），当用户编辑复盘内容后自动触发同步 | 必须，建议防抖 1000ms-3000ms |
| FR4 | 前端 UI 顶部或底部状态栏显示当前的同步状态（未保存、保存中、已保存、保存失败） | 重要 |
| FR5 | 如果同步失败，系统应在本地保留最新数据，并在网络恢复后重试或提示用户重试 | 必须 |

### 4.3 非功能需求
- **性能 (Performance)：** 同步请求不得阻塞主线程操作，用户在输入时不能感受到卡顿。
- **可扩展性 (Scalability)：** 数据库表结构应支持未来可能添加的自定义字段或复盘模板。
- **可靠性 (Reliability)：** 即使 TiDB 断连，本地需能正常查看和编辑，不影响离线使用，等待网络恢复再同步。

## 5. 设计与用户体验

### 5.1 设计原则
- **无感同步：** 大多数情况下用户不需要主动点击“保存”按钮。
- **清晰反馈：** 当网络出错或同步失败时，必须让用户明确感知，避免造成“以为保存了其实没保存”的错觉。

### 5.2 页面结构与交互流程
- **同步状态栏：** 可以在 DailyReviewPanel 或 ReviewEditor 的边角（例如右上角）增加一个微小图标：
  - 云朵+打勾：已同步至最新。
  - 云朵+转圈：正在同步。
  - 云朵+叹号（红色）：同步失败，离线。
- **交互：** 点击状态图标可查看最后同步时间，或触发手动重试。

## 6. 技术规格说明

### 6.1 API 设计
参考知识库现有的 API 设计风格，可能需要以下（由 Rust/Tauri 提供的）IPC 或 HTTP 接口：
- `sync_daily_review` (POST / IPC):
  - Request: `date` (YYYY-MM-DD), `content` (Markdown/JSON string), `compound_stats` 等相关元数据。
  - Response: `success`, `last_updated_at`.
- `get_daily_review` (GET / IPC):
  - Request: `date`
  - Response: 对应日期的完整复盘数据对象。

### 6.2 数据库设计 (TiDB)
可能需要在 TiDB 增加或更新 `daily_reviews` 表：
- `id` (Primary Key, UUID/BigInt)
- `user_id` (如果有多用户架构)
- `date` (Date, 唯一索引，标识哪一天的复盘)
- `content` (Text/LongText)
- `metrics` (JSON, 存放量化指标)
- `created_at` (Timestamp)
- `updated_at` (Timestamp)

### 6.3 前端实现建议
- 参考 `src/features/documents/KnowledgeDocumentWorkspace.tsx` 中的 `saveLocalDocument` 和 `syncSlot` 逻辑。
- 更新 `dailyReviewStore.ts` 或对应组件，加入 `isSyncing`, `syncError` 状态变量。
- 使用 `useEffect` 监听编辑器内容的变更，并使用 lodash.debounce 或类似机制调用同步 API。
