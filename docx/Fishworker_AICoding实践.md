# AI Coding实践与方法论

FishWorker 是一个个人效能管理桌面端应用（基于 Tauri + React）。本项目的一大特色在于**高度依赖 AI Coding / Vibecoding 模式驱动**，所有的业务逻辑、UI 界面均通过细致的 Markdown 需求文档和接口文档交由 AI 编写完成。模块涵盖了知识库、四象限工作台、周计划、清单系统等。

## 1. “接口与规范先行”文档开发流

AI 在局部代码生成能力极强，但在全局工程架构规划上容易妥协。因此，必须由文档（或架构级 PRD）先行锁定接口契约。传统的 PRD 主要面向人类开发者和测试，而 AI 原生 PRD 需要兼顾宏观的业务上下文和微观的技术约束。FishWorker 遵循以下标准化结构：

PRD：

- **概述与理念**：向 AI 解释**“为什么”**要做这个功能。例如在“四象限工作台”中强调了“艾森豪威尔法则”，让 AI 理解功能的业务意图，从而在生成 UI 色调（如 Q1 红色，Q2 绿色）时符合预期心理暗示。
- **边界定义**
  - **In Scope (包含)**：明确列出需要 AI 生成的具体功能点（如 2x2 矩阵、拖拽流转）。
  - **Out of Scope (不包含)**：防止 AI 产生幻觉或过度设计（例如明确提出“本次不改变已有 UI 设计”）。
- **数据结构定义**（TypeScript Interface 示例）
- **视觉及组件树拆解** 这种“保姆级”的结构化输入，大幅减少了 AI 的幻觉，保证了代码生成的质量和一次性通过率。
- 用户故事与验收标准：以 `[ ]` Checkbox 形式列出严格的交互验收标准，作为 AI 编写测试用例或自我审查的 Check-list。
- **技术规格与约束**：在 PRD 尾部直接注入架构级约束（如：使用 Zustand hook、指定某个核心方法的签名）。

接口文档：

- **全局布局锁定**：如 `Frontend_Interfaces.md` 中严格规定了工具栏的 `ToolConfig` 接口和布局容器（左中右、上下分栏）。有了这个“插槽”规范，后续让 AI 增加任何新页面，只需要一句“遵循 ToolConfig 规范增加一个模块”。
- **数据模型演进**：在迭代版本中，显式声明 `Task` 数据模型的新增字段（`description`, `deadline`），确保 AI 在修改组件时前后端数据结构一致。

## 2. 项目架构先行

**高效的“搭积木”式模块化扩展**

- 是什么：建立高度解耦的架构和明确技术选型
- 为什么：
  - Vibecoding 的初期，为了快速实现功能，AI 往往倾向于使用最原始或最简单的技术方案（例如 FishWorker 初期大量使用了 `window.dispatchEvent` 和手动 `refreshData()`），这在功能的丰富后会导致严重的性能瓶颈和状态混乱。
  - 建立的 `Frontend_Interfaces.md` 规范，应用形成了一个高度解耦的架构（如 `ToolConfig` 接口定义了统一的侧边栏工具挂载方式）。这使得在引入“四象限工作台”或“清单系统”时，AI 能够像插入插件一样，非常顺利地完成新功能的集成，几乎没有对现有系统的破坏。





## 5. 总结流程模板

1. 定义基建：编写 `Interfaces.md`，锁定路由、全局状态规范（如 Zustand）和整体布局。
2. 场景 PRD：按功能拆分独立PRD的Markdown，包含业务意图、AC 列表和 UI 约束。
3. **隔离开发：让 AI 根据单个 PRD 生成独立的组件，然后通过基建定义的接口挂载。
4. **技术债清理**：每经历 3-4 个业务迭代，输出一份纯技术维度的重构 PRD 进行代码治理。
