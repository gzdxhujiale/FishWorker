# Matt Pocock Skills 技能说明文档

> 来源：[mattpocock/skills](https://github.com/mattpocock/skills)
>
> 这些技能由 Matt Pocock 开发，面向真实工程场景（非 vibe coding），设计为小型、可组合、适用于任何模型。

---

## 一、技能总览

本仓库共安装了 **17 个 engineering 类技能**，分为三个层次：

| 层次 | 技能 | 作用 |
|------|------|------|
| **入口/编排** | ask-matt, setup-matt-pocock-skills | 路由选择、初始配置 |
| **核心工作流** | grill-with-docs, to-spec, to-tickets, implement, tdd, code-review | 从想法到交付的完整链路 |
| **辅助工具** | prototype, research, diagnosing-bugs, resolving-merge-conflicts, wayfinder, triage, improve-codebase-architecture, domain-modeling, codebase-design | 专项能力增强 |

---

## 二、主工作流（idea → ship）

大多数开发工作遵循以下路径：

```
/grill-with-docs → /to-spec → /to-tickets → /implement → /code-review
     ↑                                              ↑
  深度访谈                                      测试驱动开发
  + 文档生成                                    + 代码审查
```

### 2.1 快速路由：`/ask-matt`

不记得用哪个技能？直接问它。它会根据你的场景推荐合适的技能或工作流。

## 三、各技能详细说明

### 3.1 `/setup-matt-pocock-skills` — 初始配置

**用途**：首次使用前运行一次，配置项目的基础设施。

**做什么**：
- 设置 issue tracker（GitHub Issues / Linear / 本地文件）
- 配置分流标签词汇（triage labels）
- 设置领域文档布局（CONTEXT.md、ADR 存放位置）

**何时使用**：安装技能后、首次使用其他技能前运行一次。

---

### 3.2 `/ask-matt` — 技能路由器

**用途**：当你不确定该用哪个技能时，向它描述你的情况，它会推荐合适的技能或工作流。

**核心概念**：
- **主流程**（idea → ship）：grill-with-docs → to-spec → to-tickets → implement
- **分支**：需要原型验证？需要跨会话？会引导你走不同路径

---

### 3.3 `/grill-with-docs` — 深度访谈 + 文档生成

**用途**：在开始编码前，通过严格的问答访谈来打磨你的想法和设计，同时自动生成文档。

**做什么**：
- 对你进行无情的追问，直到决策树的每个分支都被解决
- 使用 `/domain-modeling` 技能构建项目的领域模型
- 生成 ADR（架构决策记录）和术语表（CONTEXT.md）
- 建立团队与 AI 之间的"共享语言"

**何时使用**：
- 有新功能想法，但还没想清楚细节
- 需要对现有代码做重大变更
- 想留下决策记录，而不只是代码

**与 `/grill-me` 的区别**：grill-with-docs 会留下文档痕迹（CONTEXT.md、ADR），适合有代码库的项目。

---

### 3.4 `/to-spec` — 生成规格文档

**用途**：将当前对话上下文转化为一份规格文档（PRD），发布到 issue tracker。

**做什么**：
- 综合对话中已讨论的内容（不额外访谈）
- 使用项目领域术语撰写规格
- 规划测试接缝（seams）
- 发布到 issue tracker 并标记 `ready-for-agent`

**何时使用**：经过 grill-with-docs 讨论后，想把结论固化为可执行的规格。

---

### 3.5 `/to-tickets` — 拆分为工单

**用途**：将规格、计划或对话拆分为一组"示踪弹"式工单（tracer-bullet tickets）。

**做什么**：
- 每个工单是一个垂直切片（vertical slice）
- 声明阻塞边（blocking edges）——即哪些工单必须先完成
- 在本地 tracker 中每个工单一个文件；在 GitHub/Linear 中创建原生阻塞链接
- 已完成阻塞的工单可立即被认领

**何时使用**：规格已确定，需要拆分为可独立执行的任务。

---

### 3.6 `/implement` — 实现工作

**用途**：根据规格或工单描述进行实际编码。

**做什么**：
- 使用 `/tdd` 在预定的接缝处进行测试驱动开发
- 定期运行类型检查
- 定期运行单个测试文件
- 最后运行完整测试套件
- 完成后使用 `/code-review` 进行代码审查
- 提交到当前分支

**何时使用**：工单已就绪，开始编码实现。

---

### 3.7 `/tdd` — 测试驱动开发

**用途**：以红-绿-重构（red-green-refactor）循环进行测试驱动开发。

**核心原则**：
- **红**：先写一个失败的测试
- **绿**：让测试通过（最少代码）
- **重构**：在测试保护下优化代码

**关键概念**：
- **接缝（Seam）**：测试只在公共边界进行，不测内部实现
- 好的测试读起来像规格说明，不关心内部结构
- 在写测试前先与用户确认要测试的接缝

**何时使用**：构建新功能、修复 bug、需要集成测试时。

---

### 3.8 `/code-review` — 双轴代码审查

**用途**：对某个固定点（commit/branch/tag）以来的变更进行代码审查。

**两个审查轴**：
- **Standards（标准）**：代码是否符合项目的编码规范？
- **Spec（规格）**：代码是否忠实实现了原始 issue/PRD 的要求？

**工作方式**：两个轴作为并行子代理运行，互不污染上下文，最后汇总结果。

**何时使用**：
- 审查分支或 PR
- 审查进行中的变更
- 用户说"review since X"

---

### 3.9 `/prototype` — 构建原型

**用途**：构建一次性原型代码来回答设计问题。

**两种模式**：
- **逻辑原型**（LOGIC.md）：构建小型交互式终端应用，验证状态机或业务逻辑
- **UI 原型**（UI.md）：在同一页面生成多个截然不同的 UI 变体，通过 URL 参数切换

**何时使用**：
- 不确定某个状态模型或逻辑是否"感觉对"
- 想探索 UI 应该长什么样
- 需要在纸上推理困难的场景

---

### 3.10 `/research` — 调研

**用途**：针对某个问题，基于高可信度的一手资料进行调研，并将结果保存为 Markdown 文件。

**做什么**：
- 启动后台代理进行调研
- 只使用一手资料（官方文档、源码、规范、第一方 API），不用二手文章
- 每个结论都引用来源
- 保存到仓库中约定位置

**何时使用**：
- 需要调研某个技术话题
- 需要收集文档或 API 事实
- 想将阅读工作委托给后台代理

---

### 3.11 `/diagnosing-bugs` — 调试诊断

**用途**：面对困难 bug 和性能回归时的系统化诊断循环。

**核心原则**：
- **建立反馈循环是关键**——一个能通过/失败的紧密信号
- 如果有紧密的反馈信号，二分法、假设测试、插桩都能找到原因
- 如果没有反馈信号，再多的代码凝视也救不了你

**流程**：
1. 建立反馈循环（最重要）
2. 二分定位
3. 假设检验
4. 插桩观测
5. 修复
6. 回归测试

**何时使用**：用户说"诊断"/"调试这个"，或报告某些东西坏了/报错/失败/变慢。

---

### 3.12 `/resolving-merge-conflicts` — 解决合并冲突

**用途**：解决进行中的 git merge/rebase 冲突。

**流程**：
1. 查看当前 merge/rebase 状态和冲突文件
2. 找到每个冲突的一手来源（commit message、PR、原始 issue）
3. 逐个 hunk 解决，保留双方意图，不发明新行为
4. 运行自动化检查（类型检查、测试、格式化）
5. 完成 merge/rebase（永不 `--abort`）

**何时使用**：遇到 git merge 或 rebase 冲突时。

---

### 3.13 `/wayfinder` — 大型工作规划

**用途**：规划超出单个 agent 会话能承载的大型工作。

**核心思想**：
- 将大型模糊的目标转化为 issue tracker 上的"决策地图"
- 每个决策工单解决一个具体问题（不是构建切片）
- 逐个解决直到路线清晰
- 地图是索引，不是存储——只指向工单，不重复内容

**何时使用**：
- 工作太大，一个会话装不下
- 从当前到目标的路径不清晰
- 需要先做决策再动手

---

### 3.14 `/triage` — Issue 分流

**用途**：将 issue 和外部 PR 通过状态机进行分流处理。

**做什么**：
- 分类（categorise）
- 验证（verify）
- 必要时追问（grill）
- 编写 agent-ready 的简报

**角色状态机**：每个 issue 经过一系列角色流转，最终成为可执行的工单。

**何时使用**：issue tracker 上有新 issue 需要处理时。

---

### 3.15 `/improve-codebase-architecture` — 架构改进

**用途**：扫描代码库，发现"深化机会"（deepening opportunities），生成可视化 HTML 报告。

**做什么**：
- 使用 `/codebase-design` 的设计词汇扫描架构
- 识别浅层模块（shallow modules）并提出深化重构方案
- 生成可视化 HTML 报告
- 让用户选择要处理的项目，然后进行深度访谈

**目标**：提高可测试性和 AI 可导航性。

**何时使用**：每隔几天运行一次，防止代码库退化为"泥球"。

---

### 3.16 `/domain-modeling` — 领域建模

**用途**：主动构建和锐化项目的领域模型。

**做什么**：
- 挑战术语：对照术语表验证每个用词
- 发明边缘场景：用极端情况压力测试模型
- 记录决策：将结晶的术语和决策写入 CONTEXT.md 和 ADR

**与"读 CONTEXT.md"的区别**：这个技能是**主动改变**模型，而不只是消费它。

**何时使用**：
- 需要确定领域术语或统一语言
- 需要记录架构决策
- 其他技能需要维护领域模型时

---

### 3.17 `/codebase-design` — 代码库设计词汇

**用途**：提供设计深层模块的共享词汇和纪律。

**核心词汇**：
- **模块（Module）**、**接口（Interface）**、**深度（Depth）**
- **接缝（Seam）**、**适配器（Adapter）**、**杠杆（Leverage）**、**局部性（Locality）**

**核心原则**：
- 删除测试：如果删掉一个模块，其他模块是否会变简单？
- 接口即测试面：接口就是测试的边界
- 一个适配器 = 假设的接缝，两个 = 真实的接缝

**何时使用**：设计或改进模块接口、决定接缝位置、让代码更可测试和 AI 可导航时。

## 四、推荐工作流总结

### 场景 A：有新功能想法
```
/grill-with-docs → /to-spec → /to-tickets → /implement → /code-review
```

### 场景 B：有 bug 需要修
```
/diagnosing-bugs → /tdd（修复） → /code-review
```

### 场景 C：代码库变乱了
```
/improve-codebase-architecture → 选择项目 → /implement
```

### 场景 D：大型模糊目标
```
/wayfinder → 逐个解决决策工单 → /implement
```

### 场景 E：合并冲突
```
/resolving-merge-conflicts
```

### 场景 F：不确定该用什么
```
/ask-matt → 描述情况 → 获得推荐
```

## 五、注意事项

1. 这些技能原为 Claude Code 设计，但 SKILL.md 格式与 Qoder 兼容，可直接使用
2. 首次使用前建议运行 `/setup-matt-pocock-skills` 进行初始配置
3. 部分技能（如 `/triage`、`/to-spec`）依赖 issue tracker 配置
4. 所有技能都可以通过斜杠命令（如 `/tdd`）直接调用
