C1: Store 职责过重 — 业务逻辑与持久化纠缠

Top Recommendation

涉及文件

dailyReviewStore.ts dailyReviewTypes.ts

问题

dailyReviewStore.ts 211 行，承担了过多职责：

  • CRUD 操作：saveReview / deleteReview / syncAllFromDB
  • 复杂业务计算：getCompoundStats（连续打卡、复利值、最长连续）
  • 数据验证：isReviewEmpty（判断内容是否为空）
  • 日期工具：getDaysDifference
  • 本地缓存：saveLocal / loadLocal
后果：无法单独测试复利计算逻辑；修改打卡规则需要动 Store 核心；新增统计维度会让 Store 膨胀。

建议

提取纯函数模块 dailyReviewStats.ts：

  • 将 getCompoundStats、getDaysDifference、isReviewEmpty 提取为纯函数
  • 输入: DailyReview[]，输出: CompoundStats
  • 100% 可测试，无副作用，无 Store 依赖

  • Testability：可直接对 stats 函数写单元测试

C2: ReviewEditor 状态管理复杂度高 — 4 个 ref 的隐式状态机

涉及文件

ReviewEditor.tsx

问题

组件用 4 个 ref 追踪保存状态，形成隐式状态机：

lastSavedContent <--> currentContent
lastSavedRating  <--> currentRating
currentDate (用于检测日期切换，触发"保存上一个日期")
  • 日期切换时的"自动保存前一日期"逻辑分散在多个 useEffect 中
  • isHighFreq 参数在组件内决定 debounce 时长，语义不清晰
  • 大量 inline style（~30 处），与 CSS 文件脱节
风险：快速切换日期时可能丢失内容；useEffect 依赖链难以推理。

建议

  • 提取 useReviewAutoSave hook — 封装"切换日期时保存 + debounce 自动保存"逻辑
  • 将 inline style 迁移到 CSS 类
  • 简化为单一 useEffect 监听 [date, content, rating]

C5: syncAllFromDB 合并策略与项目其他模块不一致

涉及文件

dailyReviewStore.ts mission/MissionStore.ts

问题

Daily Review 使用自定义的 syncAllFromDB（手动合并 local + DB），而 Mission 模块使用 createSyncEngine（keyed debounce 乐观同步）。两种模式并存增加认知负担。

建议

评估是否统一为 createSyncEngine 模式。但注意 Daily Review 的"批量加载 + 按日期合并"语义与 Mission 的"逐条乐观更新"不同，强行统一可能引入复杂度。

注意：此项优先级较低，建议在 C1-C3 完成后再评估。
